Threats

    - Enter isolated mode with multiple borrows
        - Make sure isolation "rules" are respected (only 1 borrow possible when user has an isolated borrow)
    - First depositor attacks / share inflation attack
        - check report from original Kamino lend for ways to donate
        - everybody_can_withdraw rule
    - Rounding directions
        - Break `ctoken_ratio` in `reserve.move` by making `1 ctoken for 0.99 underlying token`
        - Wrong use of functions that round up or down e.g. `usd_to_token_amount_lower_bound` & `usd_to_token_amount_upper_bound`
    - Boundaries are always enforced. For instance, it should not be possible to push the utilization rate over 1e18 → 100% (allowed bounds are 0 to 1e18) → this can potentially be triggered by inflating the value of `borrowed_amount` in `reserve.move`
    - Deposit & borrow same asset cannot be possible
    - Manipulating other users’ obligations should not be possible (unless it is expected e.g. liquidation)
    - Incorrect obligation’s prices updates. For instance in `obligation.move::borrow()`, the `borrow.market_value` is increased by `borrow_market_value` however, the actual market value of the borrow might have fluctuated, resulting in the `market_value` field of the `Borrow` being wrong
    - monotonicity of liquidations and withdraws (possibly others)

Properties

Health of the Obligations
    The only way for an obligation to become insolvent is by accrual of interest (over time) and change of price. Not by user actions.
            require !obligation.is_liquidatable(e); // is_healthy
            f(e, args);
            assert !obligation.is_liquidatable(e);
        - First prove that oracle price cannot return two different values for the same block. And there's no interest within the same block
        - We'll need to prove correctness of the internal accounting, e.g., 
            - obligation.deposited_value_usd == sum(deposit value * open ltv) for all deposits
            - allowed_borrow_value_usd == sum(deposit value * close ltv) for all deposits
            - similarly for unweighted_borrowed_value_usd, weighted_borrowed_value_usd
    
    No collateral => no borrow
        Obligation.deposits is empty => Obligation.borrows is empty

	Healthy obligations cannot lose collateral to became liquidatable
        - only obligation.withdraw can decrease the collateral
        - must be healthy after (obligation.is_healthy() )
    
    Integrity of is_healthy check
        - is_healthy => is_solvent
        - is_healthy with X collateral => is_healthy with >X (in the same state)
        - LTV must be monotonically increasing with debt assets and monotonically decreasing with collateral assets.
        - The solvency check should never revert.
        - For zero debt the solvency status must be true.
        - The calculation of the assets (debt and collateral) value must probe the price oracle in the latest available time stamp.

    Additional / low level / low impact
        - If there's not activity, elapsing of time must not decrease the LTV of an obligation
            - can be recalculated via obligation.refresh()

        - Changes of market price should be able to affect solvency status
            require obligation.isHealthy;
            f(e, args); 
            satisfy !obligation.isHealthy;

        - The accrued interest of a debt must be monotonically increasing with debt value (for any fixed time period).
        - Increase/decrease of the debt asset (with constant collateral asset price) must increase/decrease the LTV of a obligation.
        - Increase/decrease of the collateral asset (with constant debt asset price) must decrease/increase the LTV of a obligation.
        - Evaluation of an asset by a price oracle must be monotonically increasing with respect to the token amount (and zero for zero amount). The evaluation must not be capped.
            - i.e. no overflows there

        - Interest model should be user-agnostic (it cannot depend on the user bare address).
        - Interest model (if mutable) is controlled only by a trusted party (admin / owner / governance).
        - Given the option, transition between different interest models for a user must not decrease the debt value.
        - Solvency and liquidation thresholds are controlled only by a trusted party (admin / owner / governance).
        - For a stale obligation, If the price of the debt and collateral asset didn’t change during a block, the LTV of a obligation mustn’t change.
        - Borrowing x additional tokens must instantaneously increase the debt amount of an obligation by x.
        - The evaluation of an asset at time T must take into account the latest price possible at that time.

Solvency of the contract
    The sum of all obligation collateral amounts of some asset must be covered by the protocol reserves of the same asset
        For each Reserve: reserve.total_supply() ≥ reserve.ctoken_supply, except for method forgive()

    - Any increase to an obligation collateral must involve an increase of the protocol reserves of that same asset by the same amount.
    - Only the borrower can increase its obligation collateral (by depositing it in the contract).
    - Only the borrower or liquidator can decrease the obligation collateral, by repaying debt.
    - The protocol cannot send away more collateral tokens than the difference between its balance and sum of obligations collateral.

    Solvency Ratio
	- Ratio between underlying assets (liquidity reserve) and cTokens (collateral reserve) should only increase
	- only forbid() can decrease ratio of underlying assets and shares

Liquidations
    - Liquidation must revert for a healthy obligation.
    - Liquidation of an unhealthy obligation must succeed: 
        (under these conditions)
            - The borrower’s obligation is unhealthy (LTV has surpassed the liquidation threshold)
            - The liquidator has enough assets to pay the (partial) debt
            - The liquidation amount is less or equal to the accrued debt
            - The obligation collateral is non-empty
        - let's also check that there's always some time period for the liquidations to happen, i.e. single tick of accrue interest should never cause a solvent address to become bad debt.

    - Liquidations are profitable for the liquidator
        - First let's prove that the return values of liquidate() are correct, i.e., it returns  
            - The amount of ctokens withdrawn from the collateral reserve.
            - The actual amount of the borrowed asset that was repaid
            Let's show that the balances change accordingly as well as internal accounting. (data on Obligation and Reserve, e.g. the collateral amount subtracted from the obligation must be equal to the number of collateral tokens the protocol sends to the liquidator excluding fees)
        - Then we can just prove that the returned value of ctokens is more valuable than the returned value of "repaid"

    - Liquidation must reduce both the collateral amount and debt amount of the obligation.
        - just check that the return value are not zero
    - Any liquidator must pay a non-zero amount of debt to the protocol
        - the protocol debt asset balance must increase.

    Additional / low level / low impact
    - Liquidation of a obligation should never involve tokens other than the debt and collateral tokens of the obligation (no transfer of unrelated tokens).
    - The collateral amount received for any paid debt token must be monotonically increasing with respect to the liquidated amount.
        - Checks for overflows
    - No liquidation can take more collateral amount of that of the obligation. The protocol cannot transfer more than the obligation underlying collateral.
        - should be implied by integrity of Liquidation
    - No liquidation can succeed and pay the protocol more than the total debt of the liquidated obligation.
        - should be implied
    - The only way to DoS a liquidation is either a front-running **liquidation of the same obligation,** or if some debt was paid.
    - Liquidation of x > 0 tokens must succeed if any liquidation of y > x tokens succeeds.
    - A liquidator should never pay more than his specified bound.
        - return value <= repay_amount
    - It should never be more profitable to liquidate itself rather than just repay.
	- Liquidation lowers the LTV

Cross-contract accounting and other properties, invariants for valid state
    - Obligations data synced with Reserve data, 
        - reserve.borrowed_amount == sum of all relevant borrows in all obligations (relevant = borrow.reserve_array_index == index of this reserve)
        - ctoken_supply >= sum of all relevant deposits
    - no entering isolated mode with multiple borrows
    - they use hard-coded indexes to reference reserves, so reserve's index should never change.
    - uniqueness of reserves' assets and obligation.deposits, obligation.withdraws - there must not be two elements with the same assets
